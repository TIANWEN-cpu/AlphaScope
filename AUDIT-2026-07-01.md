## 一、严重 (CRITICAL) — 必须修复

### C1. API 认证默认关闭 — 所有端点无鉴权

**文件:** `backend/api/main.py:134-138`

`ALPHASCOPE_LOCAL_API_TOKEN` 未在 `.env` 中设置，`bool("")` 为 `False`，导致 `_is_local_token_required()` 对所有请求返回 `False`。30+ 个 API router 完全无认证，包括：
- `POST /api/datasources/credentials` — 保存 API 密钥
- `DELETE /api/datasources/credentials/{name}` — 删除密钥
- `POST /api/datalake/query` — 任意 SQL 执行
- `POST /api/tickflow/preview` — SSRF
- `POST /api/files/upload` — 文件上传
- `POST /api/chat/stream` — 任意 LLM 调用

此外，即使设置了 token，所有 GET 请求也绕过认证 (`SAFE_METHODS = {"GET", "HEAD", "OPTIONS"}`)，暴露 `/api/conversations`（所有 AI 对话）、`/api/datasources/credentials`（密钥列表）、`/api/audit`（审计日志）等敏感端点。

### C2. CORS 全开放 + 凭证传递

**文件:** `backend/api/main.py:68-72, 126-129`

`.env` 设置 `ALPHASCOPE_ALLOW_ALL_CORS=1`，导致 `allow_origins=["*"]` + `allow_credentials=True` + `allow_methods=["*"]` + `allow_headers=["*"]`。任何域可发认证跨域请求。

### C3. 主加密密钥明文存在于 release 目录

**文件:** `.env` 第 1 行

`AI_FINANCE_MASTER_KEY=iLE9MU3aLJeJo6zR_y6w8eODjCiRUpnM2HTudawnLIA` 明文存储。该密钥是 `key_vault.py` AES-GCM/XOR 加密所有 API 密钥的根密钥。虽 `.env` 已在 `.gitignore` 中（未被 git 跟踪），但该文件存在于 release 目录中，打包/分发/备份时会泄露。

### C4. TickFlow 预览端点 SSRF

**文件:** `backend/api/tickflow.py:100-116` + `backend/providers/http_json_provider.py:285-350`

`POST /api/tickflow/preview` 接受任意 URL 并从服务器发起 HTTP 请求，仅校验 URL 以 `http://` 或 `https://` 开头。无内网 IP 过滤（127.0.0.1, 10.x, 172.16-31.x, 192.168.x）、无云元数据端点过滤（169.254.169.254）、无 URL 长度限制、无 header 注入防护。攻击者可扫描内网、获取云实例凭证。

### C5. 数据湖 SQL 注入 — 可读取服务器任意文件

**文件:** `backend/quant/datalake.py:275`

用户提交的 SQL 直接拼入 DuckDB 查询：`f"SELECT * FROM ({sql.rstrip(';')}) AS _q LIMIT {lim}"`。`is_select_only()` 使用黑名单校验，但未拦截 DuckDB 文件读取函数 `read_parquet()`、`read_csv_auto()`、`read_json_auto()`。攻击者可构造 `SELECT * FROM read_csv_auto('C:/Windows/System32/drivers/etc/hosts')` 读取服务器任意文件。

### C6. 解密后的 API 密钥注入进程环境变量

**文件:** `backend/datasource_config.py:257, 443`

解密后的 API 密钥写入 `os.environ`，通过进程检查（`/proc/self/environ`）、错误消息、崩溃转储、子进程均可获取明文密钥。

### C7. `cryptography` 未声明为依赖 — 加密静默降级为 XOR

**文件:** `backend/security/key_vault.py:20-25, 85-87`

`cryptography` 库未在任何 requirements 文件或 `pyproject.toml` 中声明。缺失时 `key_vault.py` 回退到 XOR "加密"（`xor:` 前缀），XOR 与 SHA-256 派生密钥可被轻易破解。`config/safety.yaml` 声明 `api_key.encrypt_at_rest: true`，但实际加密强度取决于 `cryptography` 是否被传递性安装。PyInstaller 打包的 exe 同样遗漏此依赖。

## 二、高危 (HIGH) — 建议尽快修复

### H1. TaskQueue 竞态条件 — 无锁共享状态

**文件:** `backend/task_queue.py:89, 137-138, 98, 114, 192`

`_futures` dict 和 `_cancelled` set 的读写均无锁保护。`submit()` 在主线程写入，`_run_task()` 在工作线程读写，`cancel_task()` 在另一线程写入。可导致 `RuntimeError: dictionary changed size during iteration` 或取消操作丢失。

### H2. runtime/task_queue.py 竞态条件

**文件:** `backend/runtime/task_queue.py:86, 90-104, 110, 129`

`submit()` 用锁保护插入，但 `_run_task()` 读取和修改 task 对象（status, started_at, output_data, completed_at）时无锁，而 `get_task_dict()` 和 `list_tasks()` 同时读取。API 可观察到部分更新的任务状态（如 status=COMPLETED 但 completed_at=0）。

### H3. 绕过数据库锁直接访问连接

**文件:** `backend/task_queue.py:74, 96, 142, 165, 200` + `backend/agent_store.py:67-68`

`Database` 类有 `_db_lock` 保护，但 `task_queue.py` 和 `agent_store.py` 通过 `db._conn` 直接操作 SQLite，绕过锁。并发写入可导致 `database is locked` 错误或数据损坏。

### H4. 单例双重检查锁定缺陷

**文件:** `backend/storage/db.py:31-42`, `backend/task_queue.py:44-50`, `backend/pipeline.py:41-50`

`__new__` 用锁保护实例创建，但 `__init__` 中 `_initialized` 的读写无锁。两个线程同时调用时，第二个线程可能在第一个线程完成初始化前就看到 `_initialized = False` 并重复初始化。

### H5. 资源泄漏

| 问题 | 文件 | 说明 |
|------|------|------|
| requests.Session 未关闭 | `backend/fund_flow.py:161` | 每次调用创建新 Session，未 `.close()`，泄漏 TCP 连接池 |
| ThreadPoolExecutor 未关闭 | `backend/task_queue.py:56`, `backend/runtime/task_queue.py:54` | 无 `shutdown()` 调用，进程退出时线程可能未正确 join |
| OpenAI 客户端缓存未关闭 | `backend/models/provider_gateway.py:531-551` | `clear_client_cache()` 替换 dict 但未 `.close()` 旧客户端，泄漏 HTTP 连接池 |
| 7 个 provider 每次调用创建新 requests.get | `finnhub_provider.py:41`, `eastmoney_provider.py:44`, `fred_provider.py:59`, `hkex_provider.py:40`, `sec_provider.py:42`, `stocktwits_provider.py:37`, `wikipedia_views_provider.py:40` | 应复用 class-level Session |

### H6. 大量异常被静默吞没

全后端发现 **73 个 `except: pass` 块** 和 **174 个返回空值（`[]`, `{}`, `None`, `False`）的 `except Exception` 块**，无任何日志。关键位置：

| `backend/portfolio_optimizer.py` | 292 | 组合优化错误静默 |

### H7. `_safe()` 反模式重复 5 次

**文件:** `backend/news_data.py:32-37`, `backend/fund_flow.py:64-68`, `backend/fundamentals.py:102`, `backend/providers/akshare_provider.py:22-26`, `backend/performance_report.py:175`

## 三、中危 (MEDIUM) — 应排期修复

### M1. pyproject.toml 缺少核心依赖声明

`pyproject.toml` 仅有 `[project.optional-dependencies]`（api/rag/dev/all），无 `[project.dependencies]`。`pip install .` 安装零核心依赖。streamlit、pandas、numpy、openai、pydantic 等仅存在于 `requirements-core.txt`，对 `pyproject.toml` 不可见。

### M2. 重量级可选库列在 "core" requirements 中

`requirements-core.txt` 包含 langgraph、mlflow、optuna、prefect、skfolio、riskfolio、evidently、ragas 等 20+ 个重型库（均以 `>=` 松散版本），但代码中均有 import guard 可降级。应移至 extras（如 `[quant]`、`[mlops]`），避免核心安装过重。

### M3. Dockerfile 非真正多阶段 + 以 root 运行

**文件:** `Dockerfile`

注释声称"多阶段构建，最小化镜像体积"，实际仅一个 `FROM`，gcc/g++ 构建工具保留在最终镜像中。无 `USER` 指令，容器以 root 运行。`.dockerignore` 未排除 `build/`、`dist/`、`installer/` 等目录。

### M4. Redis 在 docker-compose 中部署但应用未使用

`docker-compose.yml` 启动 Redis 容器并设置 `REDIS_HOST`/`REDIS_PORT`，但 `backend/cache.py` 是纯内存 `TTLCache`，全代码库无 `import redis`。浪费资源且配置误导。

### M5. ruff.toml / pytest.ini 与 pyproject.toml 配置重复

`ruff.toml` 和 `pyproject.toml [tool.ruff]` 定义完全相同的配置；`pytest.ini` 和 `pyproject.toml [tool.pytest.ini_options]` 逐字节等价。`ruff.toml`/`pytest.ini` 优先级更高，导致 `pyproject.toml` 中的配置成为死配置。

### M6. Streamlit dashboard.py — 48 个静默异常 + 直接外部 API 调用

**文件:** `frontend/dashboard.py`（4,519 行，200KB）

48 个 `except Exception: pass` 无日志，数据源失败静默消失。第 666 行直接调用 `http://qt.gtimg.cn` 腾讯行情 API 绕过后端。LLM 结果按 `llm_result_{symbol}_{days}` 存入 `st.session_state` 永不过期，浏览多个股票后内存膨胀。

### M7. React 前端 — 静默回退到模拟 K 线数据

**文件:** `apps/web/src/components/Workbench.tsx:1051, 1062, 1078, 1106`

后端行情获取失败时，图表静默回退到 `generateKlineData()`（合成/模拟 K 线），无"当前显示模拟数据"提示。用户可能在不知情下看到假数据。

### M8. React 前端 — 16 个 console.* 语句留在生产代码

`apps/web/src/lib/analysisAdapter.ts`、`dataFetch.ts`、`ReportGenerator.tsx` 等文件中有 16 个 `console.log/warn/error` 调用，无构建时移除配置。

### M9. launcher.py — os.system taskkill + 无优雅 API 关闭

**文件:** `launcher.py:296-298, 213, 404-408`

`os.system(f"taskkill /PID {pid} /T /F >NUL 2>NUL")` 有 shell 注入面（pid 虽为整数但仍应用 subprocess）。uvicorn 在 daemon 线程中运行，退出时无 `server.shutdown()`，在途请求被截断。

### M10. 零前端测试

`frontend/` 目录下 16 个 Python 文件（含 200KB 的 `dashboard.py` 和 15 个组件），无任何测试文件。前端回归只能靠人工发现。

### M11. 测试覆盖重大缺口

| 模块 | 未测文件 |
|------|---------|
| storage/ | `backend.py`, `schema_upgrade.py`, `repositories/` |
| providers/ | 17 个 provider 中 11 个无测试 + 全部 4 个商业 provider 无测试 |
| rag/ | `chunker.py`, `document_pipeline.py`, `hybrid_retriever.py`, `retriever.py`（仅 degradation 路径有测试） |
| 其他 | `llm_agents.py`, `concurrency.py`, `cache.py`, `price_fetcher.py`, `retry.py`, `provider_timeout.py`, `events/extractor.py`, `ingestion/jobs.py` |

### M12. conftest.py 过于单薄 + 单例重置有交叉污染风险

`conftest.py` 仅做 `sys.path` 操作，无共享 fixture（临时 DB、mock LLM、mock HTTP）。6 个单例类在测试中直接重置 `_instance`，部分缺 teardown，测试顺序变化可能交叉污染。15 个测试文件重复 `sys.path.insert()`。

### M13. CI 无安全扫描

`.github/workflows/ci.yml` 无 `pip-audit`/`safety`/`bandit`/CodeQL/SBOM。对处理 API 密钥的金融应用是显著缺口。

### M14. Pydantic 模型无自定义校验器

全部 schema 文件中 **零个** `@field_validator`/`@model_validator`。`ChatRequest.mode`、`AnalysisRequest.mode` 等接受任意字符串而非枚举。日期字段（`start_date`, `end_date`）为无格式校验的 `str`。`AnalysisResultData.result` 等用 `dict[str, Any]` 无结构校验。

### M15. 配置中硬编码模型 ID

`config/models.yaml` 和 `config/experts.yaml`（1,318 行）硬编码模型 ID 如 `claude-sonnet-4-5`、`gpt-5.2`、`deepseek-v4-flash`，两个文件互为重复来源。provider 重命名/退役模型时静默失败。

### M16. config/safety.yaml 与 .env 策略矛盾

`config/safety.yaml` 的 `network.blocked_hosts` 包含 `localhost`、`127.0.0.1`、`0.0.0.0`，但 `.env.example` 设 `ALLOW_LOCAL_LLM_BASE_URL=1` 且桌面构建"默认为 1"。两项策略矛盾。

48 字节，内容为 `{"detail":"There was an error parsing the body"}`（FastAPI 标准错误响应），疑为手动 curl 测试产物。未跟踪且未在 `.gitignore` 中，持续显示为 untracked。

`backend/security/key_vault.py:85-87` — `cryptography` 缺失时回退到 XOR，无认证、无安全强度。代码中有日志警告但无运行时阻断。

`backend/fund_flow.py:183` — `"ut": "b2884a393a59ad64002292a3e90d46a5"` 为公开已知 token，风险低但应可配置。

冻结在 v1.5.0，仍依赖 `framer-motion`（apps/web 已迁移到 `motion`），无任何构建/启动路径引用。应删除或移至 `docs/archive/`。

`apps/web/src/index.css:1` — `@import url('https://fonts.googleapis.com/...')`，离线环境加载失败。

### L11. 缺少顶层 ErrorBoundary

`apps/web/src/main.tsx` 无顶层错误边界，`App` 组件本身的渲染错误会显示空白页。

### L12. vite 重复声明

`apps/web/package.json` 中 `vite` 同时出现在 `dependencies` 和 `devDependencies`，版本解析歧义。

### L13. Settings.tsx 死输入

`apps/web/src/components/Settings.tsx:1720` — SSE 路径 TextField 的 `onChange={() => undefined}`，不可配置的伪装输入。

### L14. api.ts 响应解析无 content-type 守卫

`apps/web/src/lib/api.ts:43` — `await response.json()` 对所有 `response.ok` 无条件执行，非 JSON 200 响应或代理 HTML 错误页会抛 `SyntaxError` 并丢失 HTTP 状态。

`frontend/components/kline_panel.py:11` — `render()` 直接索引 `df["date"]`、`df["open"]` 等，缺列或空 DataFrame 会以 KeyError 崩溃。

### L16. FRONTEND_REPAIR_AUDIT.md 行引用过时

审计文档中的行号引用（如 `api.ts:178`）已不再对应当前代码（`api.ts` 现仅 49 行，函数已拆分到 `analysisAdapter.ts` 和 `dataFetch.ts`）。

### L17. Makefile clean 目标不可移植

`Makefile:43-46` 使用 `find ... -exec rm -rf`，在 Windows 上无 POSIX find 无法运行（项目以 Windows 为主）。

### L18. docker-compose.yml version 键过时

`version: "3.8"` 在现代 Compose 中已废弃，会产生警告。

### L19. Vite API URL 硬编码在 docker-compose.yml

`docker-compose.yml:95, 100` — `VITE_API_BASE_URL: http://localhost:8000` 构建时烘焙到 JS 中，非 localhost 客户端不工作。
